## <a name="200-march-9-2015"></a>2.0.0: 9 марта 2015 г.

[Скачать версии 2.0.0](https://download.xamarin.com/objective-sharpie/ObjectiveSharpie-2.0.0.pkg)

Цели Sharpie 2.0 — важный выпуск, улучшенная драйверов на основе Clang и средство синтаксического анализа и механизм привязки на основе NRefactory. Эти улучшенные компоненты обеспечения _большую_ более высокий уровень привязки, особенно тип привязки. Были внесены многие другие изменения, которые являются внутренними для Sharpie Objective, которое создаст множество пользовательских возможностей настраивались в будущих выпусках.

Цели Sharpie 2.0 основана на Clang 3.6.1.

### <a name="type-binding-improvements"></a>Усовершенствования типа привязки

* Блоки Objective-C теперь поддерживаются. Сюда входят анонимные и встроенных блоков и блоков с именем через `typedef`. Анонимные блоки будут привязаны как `System.Action` или `System.Func` делегаты, хотя именованные блоки будут привязаны как строгое имя `delegate` типов.

* Имеется улучшенные именования эвристический алгоритм для анонимного перечисления, которые непосредственно перед которым `typedef` разрешения в builtin целочисленный тип, такой как `long` или `int`.

* Указатели C теперь связываются как C# `unsafe` указатели вместо `System.IntPtr`. Это приводит к понятнее, если требуется включить в параметры-указатели привязка `out` или `ref` параметров. Невозможно определить, всегда ли параметр `out` или `ref`, поэтому указатель сохраняется в привязке, чтобы разрешить для упрощения аудита.

* В приведенной выше привязке указателя исключение при обнаружении ранг 2 указатель на объект Objective-C в качестве параметра. В этих случаях преобладает соглашение является и как будет привязан параметр `out` (например `NSError **error` → `out NSError error`).

### <a name="verify-attribute"></a>Проверить атрибут

Вы обнаружите, что созданные Sharpie цели привязки теперь будет сопровождаться атрибутом `[Verify]` атрибута. Эти атрибуты указывают, что делать так стоит _проверьте_ что Sharpie цели было правильным, сравнивая привязка, с помощью исходного объявления C/Objective-C (которые будут предоставляться в комментарии перед объявлением привязанного).

Проверка подлинности _рекомендуется_ для всех связанных объявлений, но вероятнее всего _требуется_ для объявления, сопровождаемых `[Verify]` атрибута. Это обусловлено тем, во многих ситуациях, не хватает метаданных в исходном коде собственного источника, чтобы определить, как лучше всего создать привязку. Может потребоваться ссылаться на документацию или комментарии к коду внутри файлов заголовков, чтобы определить оптимальные привязки.

См. в разделе [проверить атрибуты](~/cross-platform/macios/binding/objective-sharpie/platform/verify.md) документации для получения дополнительных сведений.

### <a name="other-notable-improvements"></a>Другие важные усовершенствования

* `using` операторы теперь создаются на основе типов привязан. Например если `NSURL` был привязан, `using Foundation;` также создается инструкция.

* `struct` и `union` объявления привязывается, с помощью `[FieldOffset]` прием для объединения.

* Значения перечисления с инициализаторами константное выражение будет теперь правильно привязан; Полное выражение преобразуется в C#.

* Методы с переменным числом аргументов и блоки теперь привязан.

* Теперь поддерживаются платформы, с помощью `-framework` параметр. См. в документации на [привязка собственных платформ](https://developer.xamarin.com/guides/ios/advanced_topics/binding_objective-c/objective_sharpie/#frameworks) для получения дополнительных сведений.

* Objective-C исходного кода будет автоматически обнаруженных, которого следует удалить требование прохождения `-ObjC` или `-xobjective-c` для Clang вручную.

* Clang использованию модулей (`@import`) определяется теперь автоматически, который следует удалить требование прохождения `-fmodules` для Clang вручную для библиотек, которые используют новую поддержку модуля в Clang.

* Единый API Xamarin теперь является целевым объектом привязки по умолчанию; использовать `-classic` параметр для 32-разрядных только классический API.

### <a name="notable-bug-fixes"></a>Важные исправления ошибок

* Исправить `instancetype` привязки при использовании в категорию Objective-C
* Полное имя категории в Objective-C
* Префикс протоколов Objective-C `I` (например `INSCopying` вместо `NSCopying`)
